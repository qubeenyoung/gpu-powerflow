# 20260515 FP32 General Linear-System Benchmarks

This experiment prepares large `Ax=b` inputs for general dense and sparse linear
systems. The target is nonsymmetric, non-SPD solver benchmarking rather than
Cholesky-style SPD workloads.

## Build

```bash
cmake -S exp/20260515 -B exp/20260515/build
cmake --build exp/20260515/build -j
```

The build creates:

- `check_dense_io`
- `check_sparse_io`
- `general_linear_system_io` static library
- `cpu_v0`

## Dense Generation

```bash
python3 exp/20260515/scripts/generate_dense_general_fp32.py
```

The default sizes are `n=4096`, `n=8192`, and `n=16384`. To generate only one
case while testing the pipeline:

```bash
python3 exp/20260515/scripts/generate_dense_general_fp32.py --sizes 4096
```

Each dense case uses:

- `A = randn(n, n) / sqrt(n)`
- `x_true = ones(n, dtype=float32)`
- `b = A @ x_true`
- NumPy `float32`
- fixed base seed `20260515`, with per-size seed `seed + n`
- `.npy` C-order storage, documented and loaded as row-major

Expected dense `A.npy` sizes are about 64 MiB, 256 MiB, and 1 GiB for the three
default sizes.

## Sparse Download

```bash
python3 exp/20260515/scripts/download_sparse_general.py
```

To download a single matrix:

```bash
python3 exp/20260515/scripts/download_sparse_general.py --matrix Hamm/scircuit
```

The downloader uses SuiteSparse Matrix Collection Matrix Market tarballs:

```text
https://sparse.tamu.edu/MM/<group>/<name>.tar.gz
```

It skips downloads when the archive or extracted `.mtx` file already exists and
prints the final Matrix Market path after extraction.

## FP32 IO Policy

Dense data is generated and stored directly as NumPy `float32`. The `.npy`
reader accepts little-endian `float32` arrays. Dense matrices generated by the
script are C-order row-major; `to_column_major_fp32()` returns a column-major
copy for cuSolverDN, where `lda = rows`.

Sparse data is kept in Matrix Market coordinate format on disk. The C++ loader
reads `real`, `integer`, or `pattern` coordinate matrices, rejects
`symmetric`, `hermitian`, and `skew-symmetric` banners, converts one-based
coordinates to zero-based CSR, merges duplicate entries, and stores values as
`float`.

## Check Utilities

Dense check:

```bash
exp/20260515/build/check_dense_io \
  exp/20260515/data/dense/random_general_n4096
```

Sparse check:

```bash
exp/20260515/build/check_sparse_io \
  exp/20260515/data/sparse/scircuit/scircuit.mtx
```

Both tools compute `||A*x_true - b||_2 / ||b||_2`. For sparse matrices, the
checker constructs `x_true = ones(n)` and `b = A*x_true` in memory because the
SuiteSparse Matrix Market files provide only `A`.

## CPU v0 Loader Template

`src/cpu_v0.cpp` is intentionally not a CLI. It builds as `cpu_v0` and keeps the
data-loading path straight-line:

- generate dense `A = randn(n,n) / sqrt(n)` in memory with fixed seed `20260515`
- set dense `x_true=ones`
- build dense `b=A*x_true`
- convert dense `A` to column-major for cuSolverDN-style code
- load sparse Matrix Market `A`
- set sparse `x_true=ones`
- build sparse `b=A*x_true`
- compute input residuals
- leave the next lines as the CPU solver insertion point

## Solver Integration Notes

For cuSolverDN dense LU/QR benchmarks:

1. Load `A.npy`, `b.npy`, and `x_true.npy` with `load_dense_matrix_fp32()` and
   `load_dense_vector_fp32()`.
2. Convert `A` using `to_column_major_fp32()`.
3. Pass `A.values.data()` to cuSolverDN float routines with `lda = A.rows`.

For cuSolverSP, cuSPARSE, or cuDSS sparse benchmarks:

1. Load `.mtx` with `load_matrix_market_csr_fp32()`.
2. Use `row_ptr`, `col_ind`, and `values` as zero-based int32 CSR arrays.
3. Build the benchmark right-hand side with `make_rhs_from_x_true(A, x_true)`.
4. Use `relative_residual(A, x, b)` for a host-side correctness check.
