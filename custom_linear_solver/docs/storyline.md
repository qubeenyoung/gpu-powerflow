# 연구 서사 — 전력망 조류계산을 위한 배치 GPU 직접해법

> **상태**: canonical   **갱신**: 2026-06-15
> **한 줄**: 전력망 Jacobian 의 비효율은 *front 가 작다(tiny-front)* 는 한 성질에서 나온다. 배치 정적
> multifrontal 위에 tiny-front 를 GPU 에 맞추는 일련의 specialization 을 얹는다 — 그리고 그 기여의 크기를
> 정직하게 분해한다.

## 1. 풀려는 문제

전력망 조류계산(power flow)은 Newton–Raphson 반복으로 푼다. 매 반복마다 동일한 희소 구조의 Jacobian 을
갖는 선형계를 한 번 풀어야 한다. 상정사고(contingency) 분석, 시계열·확률 시나리오, 최적화 내부 루프에서는
이런 계를 **수백~수천 개 동시에** 풀어야 한다.

따라서 과제는 분명하다 — *같은 sparsity, 다른 수치*를 갖는 다수의 선형계를 GPU 에서 **배치로** 빠르게,
그리고 충분한 정확도로 푸는 것이다.

---

## 2. 왜 어려운가 — 근원은 tiny-front

전력망 Jacobian 은 일반 행렬과 다른 정의적 성질을 갖는다. 거의 평면(near-planar) 그래프에 가까워,
nested-dissection 으로 정렬해 multifrontal 로 분해하면 조밀 부분문제(front)가 **작고**(수~수십 차원),
소거트리(elimination tree)가 **깊고 좁다**.

이 한 성질이 모든 비효율의 근원이다.

- front 가 작으니 조밀 커널이 GPU 를 못 채운다 → 점유율(occupancy)이 낮다.
- 갱신(trailing) 연산의 contraction 깊이가 얇아(thin-K, K = pivot 열 수 ≲ 20) 텐서코어가 안 먹고 메모리
  지연에 묶인다.
- 트리가 깊고 좁아 레벨마다 커널 실행·동기화 비용이 누적된다.

즉 가속은 부동소수 연산량(FLOP)이 아니라 **점유·지연·동기화**에서 나온다. 범용 multifrontal 이 이런
grid-like 행렬에서 GPU 피크의 0.004 % 만 쓴다는 보고(Spatula, MICRO'23)도 같은 진단을 가리킨다.

---

## 3. 접근의 골격

바닥(substrate)은 다른 배치 직접해법(예: cuDSS)과 공유한다 — 한 번의 기호분석으로 정적 plan 을 확정하고,
B 개 시스템을 균일 배치로 묶어, 디바이스에 상주한 채 분해·해를 replay 한다. 전력망의 대각우세 덕에
pivoting 없이도 정확하다. 이 바닥은 우리 기여가 아니라 **전제**다.

우리 기여는 그 위에서 *tiny-front 를 GPU 에 맞추는* specialization 이다. 다섯 갈래로 정리된다 — 각각
§2 의 서로 다른 귀결 병목을 친다.

## 4. 기여

### (1) 티어별 전용 커널 — front 크기로 커널이 결정된다

front 인수분해의 비효율이 *크기*에서 오므로, **front 크기를 네 구간(tier)으로 나누고 각 구간에 전용 커널을
둔다.** 세 경계는 임의로 고른 값이 아니라 각기 다른 *물리적 한계*에서 도출된다 — 그래서 어떤 front 가 어느
커널을 타는지는 **크기(와 정밀도)만의 결정적 함수**이며, 실행 시 점유율이나 옵션에 따라 흔들리지 않는다.

| tier | front 크기 | 전용 커널 (shared 전략) | 경계의 근거 |
|---|---|---|---|
| **tiny** | ≤ 32 | 한 워프가 여러 front 를 묶어 처리(sub-group packing) | **워프 폭(32)** — packing 은 한 워프 안에서만 성립. 리프의 수만 개 tiny front 를 block-per-front 로 돌리면 thread 가 놀고 block barrier 만 낸다. |
| **small** | 33 – 64 | front 전체를 shared 에 올려 제자리 인수분해 | **점유율 교차점(64)** — whole-front staging 이 `fsz²` 라, 넘으면 SM 당 블록이 급감해 panel-resident 가 이긴다. |
| **big** | 65 – shared 한계 | L/U 패널만 shared, 기여블록은 global (배치) / whole-front (B=1) | small 의 `fsz²` 점유 천장을 깨 대역폭 회복 — *배치 최대 이득*(아래 (3)). |
| **large** | 그 이상 | front 는 global 상주, 패널만 staging | **shared 용량 한계** — front 를 통째로 못 올리는 크기(float 159 / double 111). 정밀도가 높을수록 더 일찍 진입. |

**정밀도는 티어와 직교한다**: TF32 모드면 네 float 티어(small/big/large)의 갱신 GEMM 이 모두 TF32 텐서코어
(Ozaki 보정 시 FP32-band)를 탄다 — big 티어도 예외 없다. 단 big 의 panel-resident 커널은 occupancy 를
회복해 *놀던 mma 헤드룸*이 없으므로, 거기서의 TF32 TC 는 FP32 scalar 보다 느리다 — **일관된 TF32 경로를
위해 감수하는 비용**이다(빠른 FP32 경로가 필요하면 `--precision fp32` 의 scalar big 티어).

핵심은 네 커널이 *각자의 병목*에 맞춰 따로 설계됐다는 점이다. 단 하나의 비-크기 조건은 **big 티어의
배치 레짐 분기**(shared 전략) 다 — panel-residency 의 이득은 *배치 점유*에서 오므로 B=1(단일 시스템)엔 줄 게
없고, 그땐 whole-front(per-front critical-path 단축, 아래 (4)/B=1 레짐)가 이긴다. 즉 big 은 *배치=panel-resident
/ B=1=whole-front* 로 갈린다(깨끗한 B==1 경계, 점유 휴리스틱 아님 — 양쪽 다 TF32 모드면 TF32 TC).
더해 한 소거트리 레벨의 이질적 front 들을 가장 큰 것에 맞춰 일괄 처리하지 않고 **tier 별로 분리해** 실행하며,
독립적인 부분트리는 별도 스트림으로 흘려 한 스트림이 놀리는 SM 을 다른 스트림의 작업이 채운다.

### (2) 텐서코어 — 저정밀을 쓰되 정확도를 지킨다

얇은 K 의 갱신 연산은 텐서코어를 그냥 켠다고 빨라지지 않는다. 그래서 front 는 FP32 로 두고 갱신 GEMM 만
TF32 텐서코어로 태우되, 저정밀로 잃는 비트를 head/tail 2-성분 보정으로 **FP32 수준 정확도까지 복원**한다.
이 보정은 메모리에 묶인 갱신 연산의 놀던 연산 헤드룸을 쓰므로 사실상 공짜다. 이 축의 핵심은 속도가 아니라
"*저정밀 하드웨어를 쓰되 정확도를 지킨다*"는 방법론이다.

그리고 우리는 그 **크기를 정직하게 분해**한다. 텐서코어 자체의 순수 기여는 +6~9 % 로 작다. 큰 속도 덩어리는
텐서코어와 무관한 커널 엔지니어링(on-chip staging, 점유 튜닝)에서 나온다. 이 분해를 전면에 둔다 —
과장하지 않는 것이 신뢰를 만든다.

### (3) 구조적 점유 회복 — big 티어가 배치에서 가장 큰 이득

위 (1) 의 **big 티어가 왜 배치 최대 레버인지**가 이 절이다. 큰 중간 front 를 통째로 shared 에 올리면 SM 당
블록이 1~2 개로 묶여 메모리 대역폭을 굶는다. front 전체 대신 *분해에 필요한 L/U 패널만* shared 에 올리고
부피를 차지하는 기여블록은 전역 메모리에 남기면, shared 사용량이 약 3 배 작아져 SM 당 블록이 3~4 배 늘고
대역폭이 회복된다. 배치에서 가장 큰 단일 이득이며, *"텐서코어가 아니라 shared 점유가 천장"* 이라는 진단이
핵심이다.

### (4) 정밀도 — all-float 가 NR 루프에 충분하다

선행 전력망 배치 해법들은 "LU 재분해의 수치 안정성 때문에" 명시적으로 FP64 를 택했다. 우리는 분해·해 전
과정을 float(FP32 front + TF32 텐서코어 + 위의 보정)로 두어도 Newton–Raphson 외부 루프의 수렴(반복 수와
최종 잔차)이 FP64 와 **동등**함을 보인다 — 반복정련 없이. 가장 크고 조건수가 나쁜 일부 케이스에는 한계가
있어 그 경계를 함께 명시한다. 이는 선행연구의 명시적 가정을 정면으로 반증하는 결과다.

### (5) GPU 상주 — 전체 NR 반복을 하나의 그래프로

상위 조류계산 엔진과 결합하면 Jacobian 조립 → 해 → 전압 갱신 → 불일치 → 노름까지 NR 한 반복 전체를 하나의
CUDA 그래프로 캡처해 replay 할 수 있고, 반복당 호스트↔디바이스 전송은 수렴 판정용 스칼라 하나뿐이다. 선행
GPU 해법들이 스트림으로 *숨기던* 전송을 구조적으로 **제거**한다.

---

## 5. 정직한 분해

이 연구의 신뢰는 수치를 부풀리지 않는 데서 나온다. 텐서코어를 켜면 large case 에서 fp32 대비 빨라지지만,
그 이득을 분해하면 약 38 % 만이 텐서코어 자체이고 나머지 ~62 % 는 정밀도와 무관한 커널 엔지니어링이다.
공정한 best-vs-best 통제에서 텐서코어 자체의 순수 기여는 중앙값 약 1.1× 이며, 작은(≤10K) 케이스에서는
front 가 너무 작아 구조적으로 이득이 없다. 우리는 이 분해를 헤드라인 옆에 그대로 둔다.

---

## 6. 한 줄 요약

근원은 tiny-front(thin-K)다. 배치 정적 multifrontal 바닥 위에 — front 크기로 **네 티어의 전용 커널을
결정적으로** 고르고(tiny/small/big/large, 경계는 워프·점유·shared 용량에서 도출), 갱신을 **텐서코어에
태우되 정확도를 지키고**, big 티어의 **shared 점유 한계를 깨서** 배치 대역폭을 회복하고, **all-float 가
NR 루프에 충분함**을 보이며, **전체 NR 반복을 하나의 그래프로** 디바이스에 상주시킨다. 텐서코어 자체 이득은
정직하게 +6~9 % 이고 가장 큰 속도 덩어리는 커널 엔지니어링이다 — 우리는 그 분해를 숨기지 않는다.
