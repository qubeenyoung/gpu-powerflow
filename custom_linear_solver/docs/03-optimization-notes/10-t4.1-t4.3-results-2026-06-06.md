## 1. T4.1 구현과 발견

### 1.1 구현 요약

`src/factorize/mid_warp.cuh` 신규 작성. `factor_small`의 warp-per-front 패턴 (32 < fsz ≤ MID_WARP_THRESH)으로 확장.

```cpp
// 환경변수 CLS_MID_WARP_THRESH (default 0 = OFF)
static const int MID_WARP_THRESH = []() {
    const char* s = std::getenv("CLS_MID_WARP_THRESH");
    if (!s) return 0;
    const int v = std::atoi(s);
    if (v <= SMALL_THRESH) return 0;
    return std::min(v, 96);
}();
```

dispatch는 SMALL_THRESH 분기 뒤, 기존 MID_THRESH 분기 앞에 삽입. precision=FP32 한정.

### 1.2 결정적이지만 silent하게 깨지는 bug — `cudaFuncSetAttribute` 누락

**초기 증상**: case8387에서 thresh=41 부터 결과 발산 (relres 1e+5 ~ 1e+22). 한 번의 시도에서는 결정적, 여러 번 실행하면 다른 값으로 발산. 처음엔 race condition으로 의심됨.

**조사 단계**:
1. fsz 값 분포 분석 → 패턴이 fsz의 32의 배수 여부와 일치하는 듯한 가설 (틀림)
2. `__threadfence_block` 추가 → 효과 없음
3. single-lane LU로 격하 (디버그) → 동일하게 fail. 단 이 디버그 자체에 divergent `__syncwarp` UB 존재해 결과 신뢰 불가였음
4. graph mode (`CLS_INTERNAL_GRAPH=ON`)로 빌드 → `setup(batched) failed: allocation failed` 에러

**원인**: `cudaFuncSetAttribute(factor_mid_warp<float>, cudaFuncAttributeMaxDynamicSharedMemorySize, 99K)` 미호출.

- factor_mid_warp는 `warps_per_block * max_fsz² * sizeof(float)` 만큼의 dynamic shared 요청
- max_fsz=42, warps=8 → 56 KB. **sm_86 기본 dynamic shared limit = 48 KB**.
- 따로 opt-in 없으면 kernel은 silent하게 launch 실패 또는 잘린 shared로 동작
- nograph mode에서는 launch가 "성공"하되 thread들이 본인 region을 넘어 다른 warp의 region에 spill → garbage
- graph mode에서는 capture 시점 검증이 더 엄격해 setup_batched 단계에서 명시적 allocation failure

**fix**: `setup`의 cudaFuncSetAttribute 호출 목록에 `factor_mid_warp<float>` 추가.

**교훈**: 새 kernel을 도입할 때, 같은 디렉토리의 다른 kernel이 받고 있는 attribute opt-in (cudaFuncSetAttribute, occupancy 설정 등)을 전부 미러링해야 함. 본 case는 48 KB 경계가 trigger.

| barrier stall | 高 | 高 | **0.0%** |

→ T4.1의 barrier 제거는 ncu로 명확히 확인되지만, **점유율 46% → 12%로 추락** 해 SOL SM이 절반 이하로 떨어짐. 결국 wall 이득 없음.

**원인 진단**: 한 block 안에서 8 warp가 서로 다른 front를 처리 → block runtime은 가장 큰 fsz front에 의해 결정됨. case8387 L=2 (716 fronts, fsz ∈ [2, 33]) 처럼 fsz variance가 큰 level에서는 load imbalance로 빠른 front의 warp가 대기. 동일 block 내 warp들이 같은 schedule slot을 묶어버려 SM 점유율이 1/N로 떨어지는 효과.

factor_small이 같은 패턴인데 잘 동작하는 이유: SMALL level (fsz ≤ 32)은 power-grid에서 대부분 leaf level이고 fsz가 비교적 균등 (예: case8387 bottom level은 fsz=2,3,4 대부분). 반면 mid level은 fsz 분포가 wide.

### 1.5 T4.1 결정

- **기본 OFF (CLS_MID_WARP_THRESH=0)**, env-var opt-in 유지
- USA-class workload (mid level fsz 분포가 균등)에서 thresh=48로 옵트인 시 약 −4% 가능
- case8387-class (variance 큰 level)에서는 사용 금지
- 향후: **per-level adaptive dispatch** — level의 fsz variance가 threshold 이하일 때만 mid_warp 사용. 미구현; T4.1.1로 분리

## 2. T4.3 구현과 결과

### 2.1 구현

`src/factorize/primitives.cuh`에 helper 추가:

```cpp
#if __CUDA_ARCH__ >= 800
#include <cuda_pipeline.h>
#endif

template <typename T>
__device__ __forceinline__ void stage_in_async(T* Fs, const T* F, int fsz2, int t, int nt)
{
#if __CUDA_ARCH__ >= 800 && !defined(CLS_NO_CP_ASYNC)
    constexpr size_t SHAPE = sizeof(T);
    for (int e = t; e < fsz2; e += nt) __pipeline_memcpy_async(&Fs[e], &F[e], SHAPE);
    __pipeline_commit();
    __pipeline_wait_prior(0);
#else
    for (int e = t; e < fsz2; e += nt) Fs[e] = F[e];
#endif
}
```

`factor_mid<T>`, `factor_mid_tc`, `factor_mid_tf32`, `factor_mid_warp<T>` 의 stage-in을 helper로 swap. `factor_big`은 shared stage-in이 없어 비대상. `factor_small`은 32-thread per front라 cp.async 효과 작지만 일관성을 위해 적용 가능 — 추후 측정 필요로 deferred.

빌드 토글: `CLS_NO_CP_ASYNC` 매크로 (CMAKE_CUDA_FLAGS) 로 fallback path 강제. sm_80 미만은 자동 fallback.

## 4. 발견과 후속 과제

### 4.1 새 발견

1. **`cudaFuncSetAttribute` 누락이 silent corruption을 만든다** — 새 kernel 추가 시 같은 디렉토리의 다른 kernel이 받는 attribute 전부 미러링 필요 (`docs/09` §5.1 에 추가)
2. **mid_warp의 load imbalance** — `factor_small`의 warp-per-front가 작동하는 이유는 small level의 fsz uniformity 덕분이지, 본질적인 패턴 우월성이 아님. mid level은 fsz variance가 커 동일 패턴 적용 시 occupancy 손실.
3. **per-level adaptive dispatch가 필요** — 단순 max_fsz threshold가 아니라 (max_fsz, fsz variance) 두 축으로 결정해야 함
4. **cp.async는 B=1 latency-bound 한정에서 일관된 이득** — B=64에서는 이미 latency hiding이 충분해 미미

### 4.2 다음 단계

#### T4.1.1 — Variance-aware adaptive mid_warp dispatch (소규모)

dispatcher 안에서 level의 fsz histogram 계산 → variance/(max-min)/CV가 threshold 이하일 때만 mid_warp 사용. 측정/구현 0.5일.

```cpp
// At dispatch time, after computing max_fsz:
int min_fsz = INT_MAX;
for (int q = b; q < e; ++q) {
    const int fsz = plan.h_front_ptr[plan.h_plcols[q] + 1] - plan.h_front_ptr[plan.h_plcols[q]];
    if (fsz < min_fsz) min_fsz = fsz;
}
const bool uniform_enough = (max_fsz - min_fsz) <= 8;  // tune
if (uniform_enough && MID_WARP_THRESH > 32 && ...) {
    // mid_warp dispatch
}
```

#### T4.2 — Sub-block 분할 (docs/09에서 deferred)

T4.1.1으로 case8387 mid level이 mid_warp에 안 가면 barrier stall은 그대로 → T4.2 (256 thread를 half-block 단위로 partial sync) 필요. 단 docs/09에서 언급한 위험 그대로 (HSync→syncwarp 격하의 정합성 보장).

#### T4.4 — persistent within-level (deferred 그대로)

T4.1 + T4.3로 case8387/USA 모두 9% 이내 개선. T4.4 시도 의의는 USA B=1 factor_big의 barrier stall 37% 잔존. 다음 round로 미룸.

### 4.3 권장 사용

- **기본 빌드**: cp.async ON (자동), CLS_MID_WARP_THRESH 미설정
- **USA-class workload**: `CLS_MID_WARP_THRESH=48` 환경변수로 추가 ~4% 이득 가능
- **case8387-class**: cp.async만 사용, mid_warp 비활성

## 7. 후속 실험 (2026-06-06 오후) — T4.1.1, factor_small cp.async

### 7.1 T4.1.1 variance gate

multifrontal.cu의 dispatch에 `level_spread = max_fsz - min_fsz` 계산 추가, `CLS_MID_WARP_VAR_GATE` (default 8) 이하일 때만 mid_warp dispatch. 7-run median:

| case | B | thresh | gate=8 | base | delta |
|------|--:|-------:|-------:|-----:|------:|
| case8387 | 1 | 33 | 0.376 | 0.371 | +1% (noise) |
| case8387 | 64 | 33 | 0.0326 | 0.0333 | **−2%** |
| USA | 1 | 64 | 2.99 | 3.05 | **−1.7%** |
| USA | 64 | 64 | 0.519 | 0.541 | **−4%** |

→ variance gate가 case8387 회귀(+10~+136%)는 모두 제거 (이제 ±2% noise 수준). USA B=64 thresh=64에서 -4% 이득. 이전 docs/10 §1.5의 "case8387-class에서 사용 금지" 권고를 "variance gate=8 적용 시 안전, 단 이득 미미" 로 정정.

**T4.1.1 결정**: default OFF (CLS_MID_WARP_THRESH=0) 유지. opt-in 권장 조합:
- USA-class: `CLS_MID_WARP_THRESH=64 CLS_MID_WARP_VAR_GATE=8` (-4% B=64)
- case8387-class: 적용 안 함 (이득 noise 수준)

## 8. T4.2.A — row-fused panel LU (2026-06-06 야간, 추가 실험)

### 8.1 가설

`lu_panel_factor`의 sync는 한 k 반복당 2개 (divide → 1 sync → rank-1 update → 1 sync), 총 2·nc 개. 만약 divide와 rank-1 update를 **per-row 단위**로 묶으면 (각 thread가 자신 담당 row의 divide와 update를 register-resident `lik`로 fuse) intermediate sync 제거 가능. → nc개 sync 만 남음 (-50%).

### 8.2 구현

```cpp
for (int k = 0; k < nc; ++k) {
    T piv = F[k*fsz + k];
    ...
    for (int i = k+1+t; i < fsz; i += nt) {
        const T lik = F[i*fsz + k] / piv;
        F[i*fsz + k] = lik;
        for (int jj = k+1; jj < nc; ++jj) {
            F[i*fsz + jj] -= lik * F[k*fsz + jj];
        }
    }
    __syncthreads();
}
```

### 8.3 nc 의존 성능 — gating 필요

| nc 범위 | 가설 | 실측 (clean A/B, CLS_NO_ROWFUSED_LU 토글) |
|---------|------|--------------------------------------------|
| nc≤8 (case8387) | win | case8387 B=1: −7.8%, B=64: 0~+2% |
| nc≤20 (USA mid) | 확실 win | USA B=1/64: **+13~+20% 회귀!** |

USA nc=20에서 회귀하는 이유: row-fused form은 thread당 `(nc-k-1)`개 FMA를 SERIAL하게 처리. nc=20이면 19 FMAs/row · ~17 rows/thread · 20 outer = 6460 cyc serial work. 원본 form은 이 work를 (fsz-k-1)·(nc-k-1) entries로 256 thread에 분산 → thread당 ~250 cyc serial. nc 커지면 serial work 증가가 sync 절감을 압도.

→ **nc ≤ 12 에서만 row-fused 사용** 으로 gate (`ROWFUSED_NC_MAX=12`). default ON.

## 9. 메타-결론 — 왜 sync 절감이 wall로 잘 안 옮겨지나

docs/09는 barrier stall 41%를 줄이면 wall도 비례해서 줄 거라 예측. 실측은 다름:

| 변경 | 예측 wall 단축 | 실측 wall 단축 |
|------|---------------:|--------------:|
| T4.1 (warp-per-front mid) | −16~18% | net 회귀 (load imbalance) |
| T4.1.1 (variance gated) | (T4.1과 동일) | −2~−4% (USA opt-in) |
| T4.2.A (row-fused panel LU) | −20% (sync count 절반) | 0~−8% (variance 큼) |
| T4.3 (cp.async stage-in) | −15~20% | −3~−4% (B=1) |

### 왜?

1. **barrier stall은 측정 가능하지만 critical path를 지배하지는 않음** — ncu가 보고하는 `% of issue cycles`는 instruction issue 관점이지 wall-clock의 명확한 fraction이 아님. 다른 stall (long_scoreboard, wait, mio_throttle) 들이 함께 시간을 잡아먹어, 한 stall 만 줄여도 다른 stall이 그 자리를 차지.

2. **occupancy의 가치를 과소 평가** — T4.1은 barrier를 0%로 만들었지만 occupancy 46%→12%로 추락. SM 점유율의 가치가 sync 절감보다 큼.

3. **per-thread serial work는 sync보다 비싸다** — T4.2.A에서 row-fused가 thread당 nc개 FMA를 serial로 만들면서 nc=20에서 +20% 회귀. sync 한 번이 ~10 cyc인데 FMA 19개도 ~19 cyc → 비슷한 단위지만 thread간 imbalance 때문에 실제 wall에 더 큰 영향.

4. **stage-in latency는 이미 hardware가 일부 hide** — cp.async가 만드는 latency hiding의 추가 효과는 B=1 (latency bound)에서만 명확.
