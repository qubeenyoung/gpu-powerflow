# 260612 목표 — factorize tier 최적화 계획

> **상태**: planning   **갱신**: 2026-06-12
> **한 줄**: small 상단 sub-band TC·mid/big 경계 재정립·big front tiling 세 축을, "factorize 는 어디서도 compute-bound 가 아니라 occupancy/under-fill/메모리 bound" 라는 공통 진단 위에 설계하고 측정 프로토콜·예상 결과·기각 기준을 명시한다.

---

## 원본 목표 (raw)

1. **small tier front 가속** — 17~32 에 해당하는 front 에 대해 TC 경로.
2. **mid / big 경계 재정립** — mid 경계가 커져(128 → 160) shared memory / register 요구량이 커짐. 다시 128 로 돌아가야 할지, 적합한 구간이 있을지, 혹은 경계를 선정하는 **근거를 마련** 가능한지.
3. **big front tiling** — big front 는 공유메모리 사용량이 높고 레벨에 front 가 적어 병렬성이 낮음. occupancy 를 높이기 위해 tiling 적용.

## 0. 공통 진단 — 세 목표가 공유하는 단일 사실

[`20260612_lab_meeting/factorize-bottleneck-ncu.md`](20260612_lab_meeting/factorize-bottleneck-ncu.md) 의 ncu 측정 결과가 세 목표 전부의 전제다:

| 사실 | 수치 (70K, tf32) | 출처 |
|---|---|---|
| **factorize 시간의 64% 가 big region** | L15–33, occupancy 24–30% | bottleneck §"레벨별 통합" |
| **mid/big occupancy = 1 block/SM, batch 무관** | B=1·16·64 모두 warp 23–33% | bottleneck §"분석" 교정1 |
| **occupancy 한계 = register(89–116/thread) + shared** | `__launch_bounds__(512,2)` 다이어트는 spill 로 net-negative | bottleneck §결론 |
| **TC 는 전 batch 에서 starved (big ≤7%)** | compute-bound 아님 | bottleneck §측정 |
| **B=1 = latency/under-fill, B≥16 = 메모리 대역폭** | big DRAM 1%(B1) → 26–35%(B64) | bottleneck §결론 |
| **small 은 occupancy 가 제일 높음 (48–69%)** | warp-packing 으로 한 블록에 여러 front | bottleneck §측정(small) |

**귀결**: 가속 레버는 FLOP 이 아니라 **occupancy 와 memory-traffic**. 세 목표를 이 렌즈로 다시 쓰면:

- 목표 1(small TC)은 **occupancy 가 이미 최고인 tier 에서 occupancy 를 깎는** 방향 → 기각 가능성 높음. *먼저 손익을 측정*해야 한다.
- 목표 2(경계)는 **shared 점유 ↔ blocks/SM** 의 직접 trade-off → occupancy 모델로 근거화 가능.
- 목표 3(tiling)은 **64% 시간을 쓰는 big 의 1-block/SM 을 깨는** 유일한 직접 레버 → **최우선·최고 기대값**.

> 정직성 기준(storyline §3B2, notes 54/55): "raw 1.24×" 같은 cap-inflation 아티팩트를 헤드라인으로 쓰지 않는다. 모든 비교는 **best-vs-best 통제**(같은 seed/cap, A·B 외 변수 고정)로 한다.

## 1. small tier 17–32 TC 경로 — ✗ 기각 (2026-06-12, ceiling-first)

> **판정**: §1.3 단계 1+2 측정 완료 → **기각**. 상세 [`03-optimization-notes/06-small-band-tc-ceiling-2026-06-12.md`](03-optimization-notes/06-small-band-tc-ceiling-2026-06-12.md).
> - band(fsz 17–32) = trailing FLOP 의 **2.7–5.4%**(large case) → TC ceiling < factorize 의 1%.
> - band 의 **65–89% 가 thin-K(nc<8)** → TF32 mma K=8 타일을 못 채움(median fill 25–35%). 불가 원인은 front 크기가 아니라 **fsz 와 무관한 thin-K**(M=N=uc 는 충분, K=nc 가 binding).
> - 채택 시 small 의 occupancy(48–69%, 전 tier 최고)를 1 block/SM(25%)로 희생.
> - small low-fill case 의 큰 FLOP%(21–31%)는 전체 FLOP 이 작은 데서 온 상대값 + 이미 net≈0 인 front-coarsening(§3B2)과 겹침.
>
> 예상(§1.5 base case)과 일치. 아래는 측정 전 계획 원문(이력 보존).

### 1.1 현재 상태
- `kSmallFrontMax = 32` ([`src/internal/types.hpp:20`](src/internal/types.hpp)). fsz≤32 는 `factor_small` (warp-packed scalar) 로 감.
- fsz 17–32 는 `factor_small_sg` 가 **sub_group_size=32 (warp 1개 = front 1개)** 선택 ([`src/factorize/small.cuh:135`](src/factorize/small.cuh)). 즉 이 sub-band 는 **이미 front 당 1 warp 전용** — TC mma(16×8×8) 를 warp 단위로 얹을 자리는 있다.
- 단, [`small-tier-no-tensorcore.md`](20260612_lab_meeting/small-tier-no-tensorcore.md) 는 small 전체에 대해 TC 를 기각: median front 4×4×2 는 16×8×8 타일의 3% 만 채움(97% padding), small 은 trailing FLOP 의 8.9% 뿐.

### 1.2 핵심 긴장 (이 목표가 풀어야 할 것)
17–32 sub-band 은 median(fsz=6) 이 아니라 small 의 **상단 꼬리**다. 이 band 의 GEMM 은 최대 30×30×16 으로 TC 타일을 의미 있게 채울 수 있다. **그러나**:
1. 이 band 가 small FLOP(8.9%) 중에서도 **얼마나 작은 조각인지** 미측정.
2. TC 를 태우려면 warp-packed(occ 48–69%) 를 포기하고 block-per-front(occ 25%) 로 가야 할 수 있음 → **occupancy 손실이 TC 이득을 넘을** 위험(목표 2/3 의 big 과 동일 함정).

### 1.3 분석 계획
1. **FLOP/시간 회계**: 6+1 케이스에서 `analyze_dump_fronts_path` CSV 로 fsz∈[17,32] front 의 개수·`Σ 2·uc²·nc`·factor 시간 비중을 집계. 전체 trailing FLOP·factorize 시간 대비 **상한(ceiling)** 부터 확정. *상한이 1–2% 면 목표 1 은 그 자리에서 기각.*
2. **TC 충족도**: 이 band 의 (uc, nc) 분포로 16×8×8 타일 fill-rate 분포를 그림. uc≥16 ∧ nc≥8 (타일 1개 충족) 비율이 band 의 몇 %인지.
3. **두 후보 경로 비교** (warp-packing 유지 여부가 분기점):
   - **(a) warp-local TC**: `factor_small` 안에서 sub_group_size=32 front 의 trailing 만 per-warp mma 로 — occupancy(48–69%) 유지, padding 손실 감수.
   - **(b) mid 승격**: `kSmallFrontMax 32→16` 으로 17–32 를 mid 커널(blocked TF32 TC)로 올림 — TC 효율↑, occupancy(48–69%→25%)↓. = storyline B2 front-coarsening 의 `CLS_SMALL_FRONT_MAX_16` 재현(이미 net≈0 보고됨 → **이 경로는 그 바를 넘어야** 채택).

### 1.4 연구 방향
- 1차 질문은 "가속되나" 가 아니라 **"이 band 가 가속할 가치(FLOP/시간 비중)가 있나"**. ceiling-first.
- 채택 시 우선 경로는 **(a) warp-local TC** — occupancy 자산(small 의 유일한 강점)을 지키기 때문. (b) 는 이미 기각 이력이 있어 보강 증거 없이는 제외.
- 신규 ablation 토글: `CLS_SMALL_WARP_TC` (default OFF) 로 (a) 를 leave-one-out 측정.

### 1.5 예상 결과
- **base case (예상 우세)**: 17–32 가 factorize 시간의 **<3%** → 어떤 경로든 전체 가속 <1%, **기각**. small-tier-no-TC 결론을 상단 band 으로 정량 확장하는 negative result(그 자체로 storyline §4 에 편입할 가치).
- **낙관 case**: 저-fill 케이스(8387/13K, nc 가 상대적으로 큼)에서만 (a) 가 band-local 1.1–1.3×, 전체 +0.5–1.5%. low-fill 정책(storyline §3B2)에 한정 편입.
- **리스크**: warp 당 register 증가로 small occupancy(48–69%) 가 깨지면 net-negative. → ncu warp% 를 A·B 로 같이 측정.

## 2. mid / big 경계 재정립

### 2.1 현재 상태 (목표의 "128→160" 정정)
- 경계는 이미 **하드코딩 128 이 아니라 shared 예산에서 유도**된다: `kFloatSharedFrontMax = float_shared_front_max()` = **159** (99 KiB / fsz²·4 fit), [`src/internal/types.hpp:70–76`](src/internal/types.hpp).
- mid 안에 **2-bucket split** 존재: `kMidSplitFrontMax = 64` ([types.hpp:96]) → mid-low(≤64) / mid-high(≤159). 각 bucket 은 자기 `fsz_cap` 으로 shared 를 잡아 occupancy 차등 ([`front_bucket()`](src/internal/types.hpp:107)).
- dispatch 는 런타임에 `fsz_cap²·elt ≤ 99 KiB` 로 mid 시도, 실패 시 big fall-through ([`src/factorize/mid.cuh:288–290`](src/factorize/mid.cuh)). FP64 는 elt=8 이라 fsz≈111 에서 자동 fall-through.
- 문제의 본질: fsz=159 mid 는 159²·4 ≈ **101 KB = 예산 전부** → **1 block/SM**. 큰 mid front 는 big 과 똑같은 occupancy 로 떨어진다. tier-thresholds 문서([`02-design-analysis/05-tier-thresholds.md`](02-design-analysis/05-tier-thresholds.md))는 여전히 128(81% 점유)을 sweet spot 으로 기술 — **문서와 코드(159) 가 불일치**. 이 목표가 그 간극을 닫는다.

### 2.2 분석 계획
1. **occupancy 모델**: `blocks_per_SM(fsz) = floor(99 KiB / (fsz²·elt))` × register 한도. fsz=64→6 blk, 96→2 blk, 112→2 blk, 128→1 blk, 159→1 blk (FP32). 경계가 **blocks/SM 계단을 어디서 밟는지** 표로.
2. **경계 sweep**: mid/big 분기 fsz ∈ {64, 96, 112, 128, 144, 159} 를 A·B 로 바꿔(`kFloatSharedFrontMax` 또는 신규 `CLS_MID_FSZ_CAP` 오버라이드) case별 `factor_ms/sys` 측정. B∈{1,16,64} 전부 — under-fill regime 과 memory regime 에서 최적 경계가 다를 수 있음.
3. **2-bucket 경계도 재검**: `kMidSplitFrontMax` 64 가 최적인지 동반 sweep(32/48/64/96).
4. **교차점 추출**: occupancy-보정 mid throughput = big throughput 이 되는 fsz* 를 case별로. 이 fsz* 가 "근거 있는 경계".

### 2.3 연구 방향
- 목표가 묻는 "근거를 마련 가능한가" 에 정면으로 답한다: **경계 = (shared-residency 의 staging locality 이득) 과 (blocks/SM occupancy 손실) 의 교차점**. 단일 상수가 아니라 **(case fill-pattern, B, precision, elt) 의 함수**일 가능성이 높음 → 결론은 "하나의 숫자" 가 아니라 **선택 규칙(decision rule)** 일 수 있다.
- "128 로 되돌릴지" 는 sweep 결과에 종속. 가설: **mid-high 를 159 까지 끌고 가는 건 손해** — 큰 mid front 는 occupancy 가 big 과 같으면서 shared 를 독점해 옆 front 를 배제. 128 또는 그 아래(112)가 B=1 에서 우세, B≥16 에서는 memory-bound 라 경계 민감도↓ 예상.
- 산출물: [`05-tier-thresholds.md`](02-design-analysis/05-tier-thresholds.md) 를 **측정으로 갱신**(159 vs 128 불일치 해소 + occupancy 모델 + sweep 표 + decision rule).

### 2.4 예상 결과
- mid-high 상한을 **128–144 로 낮추는 것이 B=1 에서 +2–5%**(큰 mid 가 big 로 가도 어차피 occupancy 동일, 대신 작은 mid 들의 blocks/SM 회복). B=64 는 ±1%(memory-bound, 경계 둔감).
- `kMidSplitFrontMax` 는 현 64 근처가 안정 예상(큰 변화 없음).
- 부산물: 문서/코드 경계 불일치 해소 + sm_90(228 KiB) 외삽 규칙.
- **리스크**: case 마다 fsz* 가 갈리면 단일 상수로 못 박고 size-적응 규칙이 필요 → 구현 복잡도↑. 그래도 "근거 마련"이라는 목표는 충족.

## 3. big front tiling  ★ 최우선 → ⚠️ 프로토타입 1 회귀 (2026-06-12)

> **프로토타입 1 결과** (2-커널 tiled-trailing, mid+big 일반화) — 상세 [`03-optimization-notes/07-tiled-trailing-mid-big-prototype-2026-06-12.md`](03-optimization-notes/07-tiled-trailing-mid-big-prototype-2026-06-12.md):
> - **13K/25K(mid-dom) 0.66–0.72× 회귀, 70K(big-dom) 1.007× 중립** (B=1, tf32).
> - ncu: tiled trailing **achieved warp% 6–14% < baseline big 27%** — under-fill 레벨은 front 2–10개라 타일로 쪼개도 *일감 부족*으로 달성 occupancy 가 안 오름(occupancy *한계* ≠ *달성*). big_split_2d 의 "1→9-10 block/SM"는 이론 한계였음.
> - **mid 일반화는 더 나쁨**: global panel 행 → mid 의 shared-residency 손실 > 작은 uc 타일 이득.
> - **결정적 검증**: under-fill 을 제거한 **B=64(GPU full)에서도 세 케이스 전부 0.62–0.68× 회귀(70K 포함)** → 회귀는 일감 부족이 아니라 **타일링 *고유* 비효율**. ncu: 타일마다 L/U 재-staging 으로 **DRAM +28%, launch +57%**. thin-K front 라 staging 이 지배적 비용인데 타일링이 이를 배수로 늘림.
> - **결론**: occupancy 천장(1 block/SM)은 실재하나 **tiling 은 그걸 회수하는 잘못된 도구**(winning regime 없음). B=1 latency 뿌리(깊고 좁은 etree)는 여전히 **합치는(launch/barrier 감소)** 방향으로 풀되 tiling 은 아님.
> - **재설정 (이후 검증으로 갱신, 2026-06-12)**:
>   - ✗ **launch fusion** — 병렬성을 못 늘림(etree DAG 가 정함). 게다가 줄일 launch 오버헤드는 이미 CUDA graph 가 처리. B=1 시간의 35–49%가 front≤4 인 spine(직렬 critical path)이라 kernel 기법으로 불가.
>   - ✗ **mid sync 감소(목표 2/③, warp-sync·sync-free U-solve)** — [`03-optimization-notes/08-mid-fewsync-prototype-2026-06-12.md`](03-optimization-notes/08-mid-fewsync-prototype-2026-06-12.md): warp-serialize 는 barrier 5.2→11.2 악화(0.90×), sync-free U-solve 는 neutral. **barrier 는 1 block/SM occupancy 의 *증상*이지 레버가 아님**(warp% 32% 불변). mid 도 tiling 과 같은 occupancy 벽.
>   - ◑ **ordering(critical-path 단축)** — [`03-optimization-notes/09-ordering-critical-path-2026-06-12.md`](03-optimization-notes/09-ordering-critical-path-2026-06-12.md): `reorder-lab` 는 stale(ordering 아님). 측정: ND ordering 이 B=1 factorize 를 **15–27% 좌우**(cap 아님, seed 만), parallel ND 는 **비결정적**(run 마다 ~10%). 어느 seed 도 robust 최선 아니나 **symbolic proxy `spine_work=Σ상위레벨 fsz²` 가 best/near-best 를 예측** → **best-of-k-by-proxy 선택**이 validated. **처음으로 구조벽에 즉시 반증 안 된 lever.** 단 ⚠️ B=1 한정(batch washes out), 반복-solve regime 에서만 k× analyze amortize, 이득 modest(default 대비 7–14% B=1 factor).
>   - ✓ **batch(B≥16) 처리량 + staging 트래픽 절감** — under-fill 을 B 로 채우는 정공법(분리 검증 대상).
>
> 아래는 측정 전 계획 원문(이력 보존).

### 3.1 현재 상태
- `factor_big` 는 **front 당 정확히 1 블록**: `idx = lbegin + blockIdx.x`, `grid(e-b, batch)` ([`src/factorize/big.cuh:249,314`](src/factorize/big.cuh)). multi-block-per-front tiling 은 **미구현**(big.cuh:298 주석은 "underfilled levels split across blocks" 를 언급하지만 실제 커널 경로엔 없음 — greenfield).
- block size: FP64 128 / TF32 512 / FP32 1024 thread. front 는 global-resident, L/U panel 만 shared staging.
- ncu 진단([factorize-bottleneck-ncu §"TC GEMM 세분화"], [small-tier-no-tensorcore §5(d)]): big = **1 block/SM, warp 25%, barrier stall 5.34 지배, TF32 pipe 98% idle**. trailing 의 MMA+drain 루프가 trailing 시간의 92% 인데 그 95% 가 stall. 진단 결론이 직접 처방을 줌: **"한 front 를 여러 블록으로 쪼개 barrier/latency 를 은닉하라."**

### 3.2 왜 이게 최고 기대값인가
big region = factorize 시간의 **64%**, occupancy 24–30% 로 평탄. 1 block/SM 을 깨는 것이 **register 다이어트(이미 net-negative 확인)·batch(occupancy 안 풀림) 가 못 한 유일하게 남은 레버**. §0 표 그대로.

> **3-case 프로파일 보강 (2026-06-12)** — [`20260612_lab_meeting/factorize-bottleneck-3case.md`](20260612_lab_meeting/factorize-bottleneck-3case.md):
> - 70K 에서 big = 시간 **51–60%**, occ **27%** 재확인 → 대형 case 최우선 레버 맞음.
> - **단 13K·25K 는 big tier 가 아예 없다**(max front <160) → mid-dominated(시간 **85–87%**가 mid). **big tiling 은 대형 case 전용 이득.**
> - **mid 도 big 과 똑같이 1 block/SM(warp 31–33%), TC <1.4%** — occupancy 천장이 mid·big 공통. → **목표 3 을 "big" 이 아니라 "big+mid 양 tier 의 1-block/SM 깨기"로 일반화**해야 13K/25K/70K 를 모두 커버. trailing 2D 타일링(T1)을 mid 에도 적용 가능(mid 는 이미 whole-front shared staging 이라 thread/occupancy 모델 동반 검토 — 목표 2 와 연결).

### 3.3 멀티프론탈 제약 (tiling 의 난점)
front 인수분해는 pivot column 따라 순차: panel-LU → U-solve → trailing, 각 단계 `__syncthreads` + cross-K 의존. 블록 간엔 `__syncthreads` 가 없으므로 front 를 단순 분할 불가. 후보 메커니즘:

- **(T1) trailing-only 2D 타일링** *(권장 1순위)*: 가장 큰 phase 인 trailing(uc×uc, 단계 시간의 ~절반)만 별도 grid 로 분리해 (i,j) blockIdx 로 fan-out. panel-LU/U-solve 는 1 블록 유지. 레벨당 phase 분리 launch 비용 ↔ trailing occupancy 회복. block-per-front 의존 깨므로 **level walk 를 phase-split 로 재구성** 필요.
- **(T2) cooperative-groups grid**: front 하나가 cooperative grid 를 점유, `grid.sync()` 로 phase 경계. 구현 단순하지만 점유 front 1개라 occupancy 이득 제한적.
- **(T3) persistent multi-block + global barrier**: trailing 을 블록 분할 + atomic/global barrier. 복잡, 위험.

→ **T1 부터.** trailing 이 big 시간의 47%(non-trailing 53%)이고 그 안이 1-block/SM stall 의 핵심이므로, trailing 만 occupancy 풀어도 big 의 절반을 친다.

### 3.4 분석 계획
1. **상한 계산(Amdahl)**: big trailing = factorize 의 `0.64 × 0.47 ≈ 30%`. trailing occupancy 를 25%→예: 75% 로 3× 빠르게 하면 그 30% 가 1/3 → factorize **약 1.25× 상한**. *구현 전에 이 상한부터 못 박고, 미달이면 T1 설계 재검.*
2. **regime 분리**: T1 의 이득은 **B=1 (under-fill/latency)** 에서 최대, B≥16 은 batch 가 이미 SM 채워 tiling 이 batch 와 SM 경쟁 → 이득↓. NR forward 는 시나리오 batch 라 B 다양 → **B∈{1,4,16,64} 전부 측정**, "어느 regime 에서 이기나" 를 명시.
3. **phase-split 비용**: 레벨당 launch 수 증가(분리 trailing). graph 모드(`CLS_INTERNAL_GRAPH`)에서 capture 되므로 반복 호출 오버헤드는 흡수 — graph on/off 둘 다 측정.
4. **검증**: ncu 로 trailing 커널 warp% / barrier-stall / TC% 가 실제 올랐는지. correctness 는 relres + `debug/fused-unittest` 의 multi-front 단위테스트로.

### 3.5 연구 방향
- ablation 토글: `CLS_BIG_TRAILING_TILED` (default OFF). big trailing 을 (front × tile) grid 로 분리.
- tile 크기 = TC strip(16-row M-tile) 단위로 잡아 §B1 TF32 mma 경로와 직교 유지(Ozaki/mma 그대로, grid 만 분할).
- 정직성: "1 block/SM 깨기" 가 진짜 occupancy 를 올렸는지 ncu warp% 로 증명. wall-time 만으로 주장하지 않음.

### 3.6 예상 결과
- **B=1 big trailing occupancy 25%→50–75%**, big trailing 1.5–2.5×, factorize **B=1 +10–20%**(§3.4 상한 1.25× 안).
- **B≥16 은 +0–5%** — memory-bound + batch 가 이미 SM 채움(정직한 캡). tiling 의 가치는 **single-system/소-batch NR** 에 집중.
- panel-LU/U-solve(non-trailing 53%) 는 손 안 댐 → 이게 다음 천장(small-tier-no-TC §5(c) 의 `1/(1−f)` 와 동일 구조).
- **리스크**: phase-split launch 비용이 graph-off 에서 이득 상쇄 / under-fill 레벨(front 2–3개)은 분할해도 SM 못 채움(이미 적은 front × tile 로 완화). T1 실패 시 T2 로 후퇴.

## 4. 우선순위 · 측정 프로토콜 · 공통 인프라

### 4.1 우선순위
| 순위 | 목표 | 기대값 | 근거 |
|---|---|---|---|
| **1** | **big front tiling (T1)** | **B=1 +10–20%** | 시간의 64%, 1-block/SM 직격, 처방이 ncu 에서 직접 나옴 |
| 2 | mid/big 경계 근거화 | B=1 +2–5%, 문서 정합 | sweep 만으로 결론, 저위험, decision-rule 산출 |
| 3 | small 17–32 TC | <1% (기각 유력) | ceiling-first, occupancy 손실 위험, 이미 부분 negative |

→ **3 은 ceiling 측정(1.3 단계 1)만 먼저** 하고, 상한이 작으면 즉시 negative result 로 종료(시간 절약).

### 4.2 공통 측정 프로토콜 (정직성)
- **best-vs-best**: A·B 외 모든 변수(seed=1588 또는 고정, panel-cap, `--serial-nd`) 동일. cap-inflation 금지(notes 54/55).
- **regime 전수**: 모든 결과를 B∈{1,16,64} 로. "어느 regime 에서 이기나" 를 항상 명시.
- **ncu 인과**: wall-time 개선 주장 시 그 메커니즘(warp% / barrier-stall / DRAM% / TC%)을 ncu 로 동반 입증.
- **Amdahl 닫기**: 부분 가속(s) × 그 부분 비중(f) → 전체 예측이 실측과 맞는지 확인(small-tier-no-TC §5(c) 방식).
- **케이스**: 8387 / 13659 / 3012wp / 6468rte / 25k / USA / 70K (lab_meeting 세트).
